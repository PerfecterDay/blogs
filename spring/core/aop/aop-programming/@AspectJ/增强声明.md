# 声明增强（Advice）
{docsify-updated}

`Advice` 与 `Pointcut` 表达式相关联，并在切点匹配的方法执行之前、之后或前后运行。切点表达式可以是内联切点，也可以是对命名切点的引用。

## Before Advice
可以在切面中使用 `@Before` 注解声明 before advice 。
```
@Aspect
public class BeforeExample {

	@Before("execution(* com.xyz.dao.*.*(..))")
	public void doAccessCheck() {
		// ...
	}
}
```

如果引用的是命名切点：
```
@Aspect
public class BeforeExample {

	@Before("com.xyz.CommonPointcuts.dataAccessOperation()")
	public void doAccessCheck() {
		// ...
	}
}
```

## After Returning Advice
`After returning advice` 在被增强的方法执行返回后开始执行。可以使用 `@AfterReturning` 注解：
```
@Aspect
public class AfterReturningExample {

	@AfterReturning("execution(* com.xyz.dao.*.*(..))")
	public void doAccessCheck() {
		// ...
	}
}
```
有时，需要在 `Advice` 主体中访问被增强方法实际返回的值。可以使用 `@AfterReturning` 形式将返回值绑定起来以实现此访问，如下例所示：
```
@Aspect
public class AfterReturningExample {

	@AfterReturning(
		pointcut="execution(* com.xyz.dao.*.*(..))",
		returning="retVal")
	public void doAccessCheck(Object retVal) {
		// ...
	}
}
```
`returning` 属性中使用的名称必须与 `Advice` 方法中的参数名称一致。当方法执行返回时，返回值将作为对应的参数值传递给 `Advice` 方法。 `returning` 子句还限制匹配范围，仅限于返回指定类型值的方法执行（此处为 `Object` 类型，匹配任何返回值）。

请注意，在使用 `AfterReturning advice` 时，无法返回完全不同的引用。

## After Throwing Advice
`After Throwing Advice` 在一个指定方法执行抛出异常时执行。可以使用 `@AfterThrowing` 声明：
```
@Aspect
public class AfterThrowingExample {

	@AfterThrowing("execution(* com.xyz.dao.*.*(..))")
	public void doRecoveryActions() {
		// ...
	}
}
```
通常， `Advice` 仅在抛出特定类型的异常时运行，并且在 `Advice` 主体中也需要访问抛出的异常。可以使用 `throwing` 属性来限制匹配条件（如需限制——否则请使用 `Throwable` 作为异常类型匹配任何异常），并将抛出的异常绑定到 `Advice` 中的参数。以下示例展示了具体实现方式：
```
@Aspect
public class AfterThrowingExample {

	@AfterThrowing(
		pointcut="execution(* com.xyz.dao.*.*(..))",
		throwing="ex")
	public void doRecoveryActions(DataAccessException ex) {
		// ...
	}
}
```
`throwing`属性中使用的名称必须与 `Advice` 方法中的参数名称一致。当方法执行通过抛出异常退出时，该异常将作为对应的参数值传递给 `Advice` 方法。 `throwing` 子句还限制匹配范围，仅限于抛出指定类型异常（此处为 `DataAccessException` ）的方法执行。

请注意， `@AfterThrowing` 并非表示通用异常处理回调。具体而言， `@AfterThrowing` 增强方法仅应接收来自连接点（用户声明的目标方法）本身的异常，而不应接收来自关联的 `@After/@AfterReturning` 方法的异常。

## After (Finally) Advice
当匹配的方法执行结束时， `after finally advice` 才会运行。它通过 `@After` 注解进行声明。 `after finally advice` 必须准备好处理正常返回和异常返回两种情况。它通常用于释放资源等类似目的。以下示例展示了如何使用 `after finally advice` ：
```
@Aspect
public class AfterFinallyExample {

	@After("execution(* com.xyz.dao.*.*(..))")
	public void doReleaseLock() {
		// ...
	}
}
```

请注意， `AspectJ` 中的 `@After advice` 被定义为 `after finally advice`，类似于 `try-catch` 语句中的 `finally` 代码块。它将在任何结果情况下被调用——无论是从连接点（用户声明的目标方法）的正常返回还是异常抛出，而 `@AfterReturning` 仅适用于成功的正常返回。 `@AfterThrowing` 仅在抛出异常时生效，两者合起来才是 `@After` 。

## Around Advice
最后一种增强是环绕增强。环绕增强在匹配方法的执行过程中"环绕"运行。它既能在方法执行前后执行操作，也能决定方法何时、如何甚至是否实际运行。当需要在线程安全模式下共享方法执行前后的状态时（例如启动和停止计时器），通常会使用环绕增强。

环绕增强通过在方法上添加 `@Around` 注解来声明。该方法应声明 `Object` 作为其返回类型，且方法的首个参数必须为 `ProceedingJoinPoint` 类型。在增强方法的本体中，必须调用 `ProceedingJoinPoint` 的 `proceed()` 方法才能使底层方法运行。若不带参数调用 `proceed()` ，则调用方原始参数将作为底层方法的参数传递。针对高级应用场景，`proceed()` 方法提供重载版本，可接受参数数组（ `Object[]` ）。该数组中的值将在底层方法调用时作为参数传递。

`around` 增强返回的值即为方法调用方所看到的返回值。例如，一个简单的缓存切面可以在存在缓存值时直接返回该值，若不存在则调用 `proceed()` 方法（并返回该方法的返回值）。需注意的是，在 `around` 增强的本体中， `proceed` 方法可能被调用一次、多次或根本不被调用，这些情况均属合法。

若将 `around` 增强方法的返回类型声明为 `void` ，则始终会向调用方返回 `null` ，这实际上会忽略任何 `proceed()` 调用的结果。因此 `around` 增强方法声明 `Object` 作为返回类型。增强方法通常应返回 `proceed()` 调用的返回值，即使底层方法的返回类型为 `void` 。但根据具体用例，增强方法也可选择性地返回缓存值、封装值或其他类型值。

```
@Aspect
public class AroundExample {

	@Around("execution(* com.xyz..service.*.*(..))")
	public Object doBasicProfiling(ProceedingJoinPoint pjp) throws Throwable {
		// start stopwatch
		Object retVal = pjp.proceed();
		// stop stopwatch
		return retVal;
	}
}
```

## Advice Parameters
Spring提供了完全类型的增强机制，这意味着可以在增强签名中声明所需参数（如前文返回值和抛出异常示例所示），而非始终使用 `Object[]` 数组。本节后续内容将介绍如何使参数及其他上下文值在增强主体中可用。首先，我们将探讨如何编写通用增强，使其能够获取当前所处理方法的相关信息。

### Access to the Current JoinPoint
任何增强方法都可声明其第一个参数为 `org.aspectj.lang.JoinPoint` 类型。请注意， `around` 增强必须声明其第一个参数为 `ProceedingJoinPoint` 类型，该类型是 `JoinPoint` 的子类。

`JoinPoint` 接口提供了以下方法：
+ `getArgs()`: Returns the method arguments.
+ `getThis()`: Returns the proxy object.
+ `getTarget()`: Returns the target object.
+ `getSignature()`: Returns a description of the method that is being advised.
+ `toString()`: Prints a useful description of the method being advised.

### Passing Parameters to Advice
前面已经了解了如何绑定返回值或异常值（使用 `after returning` 和 `after throwing` 增强）。要使参数值在增强主体中可用，可以使用 `args` 的绑定形式。若在 `args` 表达式中用参数名替代类型名，则当增强被调用时，对应参数的值将作为参数值传递。以下示例将更清晰地说明这一点。假设需要对以 `Account对` 象作为首参数的 `DAO` 操作执行增强，并在增强主体中访问该账户。可采用如下写法：
```
@Before("execution(* com.xyz.dao.*.*(..)) && args(account,..)")
public void validateAccount(Account account) {
	// ...
}
```

点切表达式中的 `args(account,..)` 部分具有双重作用：
+ 首先，它将匹配范围限定为仅包含满足以下条件的方法执行: 该方法至少接收一个参数，且传递给该参数的实例必须是 `Account` 类型；
+ 其次，它通过 `account` 参数使实际的 `Account` 对象可供增强使用。

另一种写法是声明一个切入点，当它匹配到连接点时"提供" `Account` 对象的值，然后在增强中引用该命名切入点。具体实现如下：
```
@Pointcut("execution(* com.xyz.dao.*.*(..)) && args(account,..)")
private void accountDataAccessOperation(Account account) {}

@Before("accountDataAccessOperation(account)")
public void validateAccount(Account account) {
	// ...
}
```

代理对象（ `this` ）、目标对象（ `target` ）以及注解（ `@within` 、 `@target` 、 `@annotation`  和 `@args` ）均可采用类似方式进行绑定。下一组示例展示了如何匹配带有 `@Auditable` 注解的方法执行，并提取审计代码：
```
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Auditable {
	AuditCode value();
}


@Before("com.xyz.Pointcuts.publicMethod() && @annotation(auditable)")
public void audit(Auditable auditable) {
	AuditCode code = auditable.value();
	// ...
}
```

## Advice Ordering
当多个切面增强都希望在同一个连接点运行时会发生什么？Spring AOP遵循与 `AspectJ` 相同的优先级规则来确定增强的执行顺序。最高优先级的增强在"进入连接点时"优先执行（因此，当存在两个 `before` 增强时，优先级较高的增强先执行）。在离开连接点时，最高优先级的切面增强最后执行（因此，当存在两个 `after` 切面增强时，优先级最高者将第二个执行）。

当两个定义在不同切面的增强都需要在同一个连接点运行时，除非另有说明，否则执行顺序是**未定义的**。可通过指定优先级控制执行顺序。这采用Spring的常规方式实现：在切面类中实现 `org.springframework.core.Ordered` 接口，或使用 `@Order` 注解标注。对于两个切面，返回 `Ordered.getOrder()` 较小值（或注解值）的切面具有更高优先级。

特定切面的每种独立增强类型在概念上都旨在直接应用于连接点。因此， `@AfterThrowing` 增强方法不应接收来自关联的 `@After/@AfterReturning` 方法的异常。

在同一 `@Aspect` 类中定义且需在相同连接点执行的增强方法，其优先级依据增强类型按以下顺序从高到低分配：`@Around` 、 `@Before` 、 `@After` 、` @AfterReturning` 、 `@AfterThrowing` 。但需注意，根据AspectJ对 `@After` 的"最后增强语义"，同一切面中 `@After` 增强方法将在任何`@AfterReturning` 或 `@AfterThrowing` 增强方法之后被调用。
